**S.O.L.I.D**



single responsibility, open-closed, liskow substitution, interface segregation, dependency inversion



SR - classes should focus / manage only 1 thing

&nbsp;  - classes should not depend on external stuff



OC - classes must be open for extensions but closed for changes

&nbsp;  - promotes inheritance



LS - design by contract = an interface is a contract between two objects

&nbsp;	e.g.: Person (abstract) extended into Student extended into a Bachelor student / Erasmus student, then extend Bachelor into		Master and that into PhD (this is a framework)

&nbsp;  	- concept used: up-casting

&nbsp;  	- even if Person is abstract, you can still define Person p to manage any object in the framework

&nbsp;	e.g.: Person p = new PhdStudent();

&nbsp;  - the code should work for every instance of derived class without affecting functionality (for example by using overriding)	



IS - interface is a class that will "convince" all clases in a framework to implement something 

&nbsp;  - e.g. software for electric vehicles:

&nbsp;	Interface Vehicle having 20+ functions vs 4 individual Interfaces having 4-6 functions each

&nbsp;		e.g. ControlVehicle, ControlLight, ControlRadio, ControlCD



DI - not depend on concrete classes (real classes)

&nbsp;  - program to interfaces, not implementations

&nbsp;  - by using interfaces, we are not required to change our class when the implemented external class changes
