Structural design patterns

\-recap last time: 

>ADAPTER - for lab test we will focus on objects adapter, composite

>DECORATOR - change objects at runtime



\-FACADE - implied complexity of libraries, projects, solution

client -> facade class with methods -> multiple classes

\-in main **create 1 single facade object** with "new" classes as parameters, and **call 1 single function** that calls multiple other functions for all classes.





COMPOSITE SCENARIO - structural tree of classes (framework)

\-start with 1 abstract class where multiple methods are declared. Some of them will be also defined in the abstract class by throwing unsupported exception.

\-create 1 abstract node class with all method from the list, but only override the ones you actually use in this class

\-create a composite class with **all** the methods listed, which extends the abstract class, and this will have to override **all** the methods.

\*The difference between the two is that only the composite class is also included in a "has-a" relationship with the parent abstract node class.
