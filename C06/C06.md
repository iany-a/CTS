Builder pattern:

* classes have too many attributes = complex objects
* risk of forgetting to set some of the attributes
* constructor for all of the attributes is too complex and prone to human error
* multiple constructors for combinations of specific attributes are not a solution either
* R: creating the device should not be an atomic operation (aka create it with all the attributes at once)

\*in this case the constructor with 7 parameters is an atomic operation as it requires operation that will not be needed)

* make setters and constructors private to force users to use the builder

\*in JAVA, in order to create a nested class, the inner class needs to be set as static

\* inner static class will have access to everything inside the outer class

* initially we set an outer class object inside the inner class: private SmartDevice smartdevice = null;
* then in all of the methods, we return the builder, not the outer object
* in main, we use OuterObject outerobject = new OuterObject.InsideBuilder().addAttribute().addAttribute2().addAttribute3()... .build();
* we can use a director to stop ourselves from copy/paste too much code for specific models



Prototype pattern: used for complex creation processes (not the object itself)

* uses the clone() method for efficiency, by reusing the existing object
* Prototype1 extends AbstractPrototype, and has method +clone().AbstractPrototype
* Prototype2 extends AbastractPrototype, and has method +clone().AbstractPrototype
* @Override clone() is being used to deep copy the object, hence "implements Cloneable" needs to be added on the class header
